# Conventions

- User-facing chat is Brazilian Portuguese; repository artifacts, comments, commit messages, prompts, and memory notes are English.
- Ruby code uses `TextAdventures` namespace; web classes live under `TextAdventures::Web`; scene classes live under `TextAdventures::Scenes`.
- Keep gameplay/domain logic transport-agnostic. HTTP, WebSocket, JSON serialization, session TTL, and state patch concerns belong under `lib/web/`.
- Prefer YAML content edits for item/shop/creature/dungeon-block changes instead of hardcoding content in Ruby.
- Runtime dungeon entity markers are generated by code; do not put enemy or loot markers directly into `data/dungeon_blocks.yml` tiles.
- Tests mirror source domains: `spec/domain`, `spec/scenes`, `spec/web`, `spec/persistence`, `spec/data`, `spec/e2e`.
- Existing tests include exact dependency assertions in `spec/project_setup_spec.rb`; update them when adding/removing Gemfile dependencies.
- Frontend is plain JavaScript and DOM/canvas code; keep it build-free unless the project intentionally adopts a build tool.
- Avoid broad abstractions when a scene/domain method can follow the existing direct style.
- Use deterministic seeds (`TEXT_ADVENTURES_RANDOM_SEED=0` or API seed payloads) for gameplay smoke checks.
- MCP workflow convention:
  - Serena is the primary repository-local code intelligence and local-memory tool.
  - Basic Memory is the durable cross-session planning, findings, decisions, handoff, and validation-evidence store.
  - Context7 is for external library/framework/API documentation and version-sensitive examples.
  - Do not store secrets or volatile logs in MCP memory; keep durable notes concise and English-only.
